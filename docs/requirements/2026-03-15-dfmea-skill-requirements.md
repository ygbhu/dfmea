# DFMEA Skill 需求文档

## 1. 文档目的

本文档定义 DFMEA skill 的业务目标、使用场景、功能边界、约束条件与验收标准，作为后续架构设计、实施计划、实现开发和评审验收的上位依据。

本文档回答“要做什么、为什么做、做到什么程度算完成”，不规定具体实现方案。

---

## 2. 背景与目标

### 2.1 背景

DFMEA（Design Failure Mode and Effects Analysis，设计失效模式与影响分析）通常以表格或分散文档方式维护，存在以下问题：

- 难以支持大规模结构化数据的分层维护
- 难以让通用 Agent 在缺少领域上下文时稳定执行增删改查
- 难以进行横切查询、批量生成和一致性校验
- 数据与操作规则分散，导致多人或多轮 Agent 操作后容易失真

### 2.2 产品目标

构建一个可供任意 Agent 使用的 DFMEA skill，使 Agent 可以基于 Markdown 文件对 DFMEA 数据进行初始化、维护、查询、校验和导出，并在可审计、可恢复、可扩展的前提下支持大规模节点管理。

### 2.3 成功目标

- Agent 可以在无人工逐步指导的情况下完成标准 DFMEA 数据操作
- DFMEA 数据在验收标准所覆盖的操作序列完成后，仍保持结构一致、引用有效、可校验
- 支持结构化分层浏览与横切查询
- 支持 10k-100k 分析层节点规模下的可维护存储
- 技能文档和数据格式对不同 Agent 保持可读、可理解、可执行

## 3. 术语与范围

### 3.1 术语

- DFMEA：设计失效模式与影响分析
- Agent：可读取 skill 文档并对本地文件执行操作的智能体
- Skill：提供操作规则、触发条件和工作流的说明文档集合
- 项目：一个独立的 DFMEA 分析对象
- 结构层：项目、系统、子系统、组件等层级结构
- 分析层：功能、失效模式、失效影响、失效原因、措施等分析内容
- 要求：功能必须满足的条件或规格，如性能指标、法规要求、设计约束
- 特性：要求的可测量或可检验表达，如尺寸公差、材料属性、电气参数
- 横切查询：跨组件、跨功能、跨失效链的查询与统计

### 3.2 需求判定定义

为避免后续架构和实现阶段对同一术语产生不同理解，本文档约定以下判定口径：

- 节点：指属于源数据范畴的、可被唯一标识、可被读取、可被修改、可被引用或可被删除控制的 DFMEA 逻辑对象。派生索引中的条目不属于节点
- 主归属关系：指一个节点在 DFMEA 结构中的唯一上级归属关系，用于形成稳定层级
- 引用关系：指节点间除主归属关系之外的受控关联，用于表达共享、传播或追溯关系
- 源数据：指作为事实来源的项目、结构、分析类数据内容
- 派生索引：指从源数据计算生成、可被重建、不作为唯一事实来源的数据集合
- 受控删除：指删除动作必须满足预设规则，并同步处理其对子节点、引用关系和一致性的影响
- 一致性校验：指对标识唯一性、归属有效性、引用有效性、孤儿数据和派生数据失真进行检查
- 可追溯：指查询、导出或校验结果能够定位回对应源数据位置或唯一标识

### 3.3 业务范围

本需求文档覆盖：

- 面向 Agent 的 DFMEA skill 能力边界
- DFMEA 数据的逻辑对象范围
- 数据维护、查询、校验、导出等核心操作需求
- 质量属性和验收标准

本需求文档不覆盖：

- UI 界面设计
- 在线多人协作平台
- 企业级权限系统
- 云端数据库或服务端 API
- AI 自动推理质量本身的评估模型
- 敏感数据脱敏、权限分级或合规审计体系

## 4. 用户与角色

### 4.1 主要用户

- DFMEA 工程师：维护结构、功能、失效链和措施
- 设计工程师：补充设计功能、约束、风险与改进措施
- 质量工程师：检查 AP、严重度、措施状态和一致性
- 项目负责人：查看项目进展、统计与待关闭措施

### 4.2 系统内角色分工

- 人类用户负责提出任务目标、确认边界、审阅结果
- Agent 负责按 skill 规则执行文件级操作、查询、校验和导出
- Git 或文件系统负责保存变更历史和审计痕迹

---

## 5. 核心使用场景

### 5.1 初始化场景

- 创建一个新的 DFMEA 项目
- 生成项目元数据、初始结构、初始计数状态和必要索引骨架

### 5.2 结构维护场景

- 新增系统、子系统、组件
- 修改结构节点的名称、描述、归属关系和元数据
- 删除空结构节点或按规则清理后删除结构节点

### 5.3 分析维护场景

- 为组件新增功能节点
- 为功能新增失效模式、失效影响、失效原因和措施
- 修改分析项的 S/O/D/AP、状态、负责人、截止日期、引用关系
- 删除分析节点并清理相关引用

### 5.4 查询与统计场景

- 按组件查看分析概览
- 按 AP、严重度、措施状态进行横切查询
- 检索最近变更
- 按关键词或 ID 搜索节点

### 5.5 校验与维护场景

- 校验主归属关系是否有效
- 校验引用关系是否有效
- 校验索引是否与源数据一致
- 重建派生索引

### 5.6 导出场景

- 将现有 DFMEA 数据导出为传统表格视图或标准 FMEA 表

## 6. 功能性需求

### 6.1 项目管理需求

- 系统必须支持创建独立 DFMEA 项目
- 系统必须支持读取项目级元数据
- 系统必须支持标识项目当前数据是否存在待重建索引状态

### 6.2 节点模型需求

- 系统必须支持以下逻辑对象：Project、System、Subsystem、Component、Function、Failure Mode、Failure Effect、Failure Cause、Action
- 每个逻辑对象必须具有稳定、唯一、不可复用的标识符
- Function 节点必须能承载功能描述、关联要求和关联特性作为结构化子信息
- 系统必须支持结构层的明确主归属关系
- 系统必须约束主归属关系的合法层级组合，不允许跨层级或违反方法论的归属
- 系统必须支持受控的跨节点引用关系

### 6.3 结构层维护需求

- 系统必须支持新增 System、Subsystem、Component
- 系统必须支持修改结构层节点的基础属性
- 系统必须阻止不符合规则的结构层删除操作
- 系统必须支持结构树的遍历与定位

### 6.4 分析层维护需求

- 系统必须支持在组件下创建 Function
- Function 必须支持记录其关联的要求和特性，以便建立从功能到失效模式的完整追溯
- 系统必须支持在 Function 下维护完整失效分析链
- 系统必须支持措施状态流转
- 系统必须支持一个措施关联多个原因的场景
- 系统必须支持跨节点引用的维护和清理

### 6.5 查询需求

- 系统必须支持按组件查看分析摘要
- 系统必须支持按 AP 等级查询
- 系统必须支持按严重度查询
- 系统必须支持按措施状态查询
- 系统必须支持关键词和 ID 搜索

### 6.6 校验需求

- 系统必须支持对主归属关系有效性进行校验
- 系统必须支持对跨节点引用有效性进行校验
- 系统必须支持发现孤儿节点、非法引用和派生数据失真
- 系统必须在删除或批量修改后支持执行一致性校验

### 6.7 异常处理需求

- 系统必须在遇到非法引用、非法删除、标识冲突或归属失效时拒绝静默通过
- 系统必须在无法安全完成写操作时返回明确失败结果
- 系统必须在派生索引过期、缺失或损坏时支持重新建立可用查询状态
- 系统必须在高影响操作失败后，至少保证源数据仍可被校验与修复
- 系统必须在异常结果中提供问题类型、问题位置和处理提示

### 6.8 输出需求

- 查询结果至少应包含节点唯一标识、节点类型、简要描述和源位置或可追溯定位信息
- 校验结果至少应包含问题级别、问题类型、受影响对象、问题位置和建议处理方向
- 系统必须支持将 DFMEA 数据导出为适合审阅的表格格式
- 导出结果至少应保持与源数据之间的可追溯映射关系
- 输出格式应便于人类审阅，也应便于 Agent 继续消费

## 7. 非功能性需求

### 7.1 Agent 友好性

- 数据格式必须为人类和 Agent 都可直接阅读的文本格式
- 操作规则必须自描述，避免依赖特定 Agent 的隐藏行为
- 常见任务必须能通过有限文件读取定位到目标数据

### 7.2 可维护性

- 数据结构必须支持长期演进和局部修改
- 单个文件不应无限膨胀到难以阅读和难以稳定修改
- 派生索引必须可重建，不能成为唯一事实来源

### 7.3 可审计性

- 关键变更必须能通过 Git 或文件变更记录进行追踪
- 数据文件与派生文件应可区分
- 删除、重建、导出等高影响操作需要可追溯

### 7.4 一致性与恢复能力

- 任意写操作后，系统应能够通过校验发现不一致状态
- 派生索引损坏时，应可通过源数据重建恢复
- ID 一旦分配，不应因删除而复用

### 7.5 性能与规模

- 结构层节点规模目标为不超过 100
- 分析层节点规模目标为 10k-100k
- 常见聚焦式读取应尽量限制在局部文件范围，不应依赖全量项目扫描
- 横切查询可依赖派生索引，不要求每次实时全量扫描，但索引失效后必须可恢复
- 全量重建类操作允许成本较高，但必须可重复执行、可中断后重新发起

### 7.6 编码与兼容性

- 所有数据文件必须使用统一文本编码，V1 以 UTF-8 无 BOM 作为目标约束
- 方案必须兼容 Windows 本地文件操作环境
- 不要求依赖数据库、服务端进程或专用索引服务

---

## 8. 约束条件

### 8.1 方法论约束

- 需求建模应与 AIAG VDA DFMEA 7 步法保持一致

### 8.2 技术约束

- V1 必须以 Markdown 文件作为主存储介质
- V1 必须以本地文件系统为运行基础
- V1 不引入数据库作为必需依赖

### 8.3 运行约束

- V1 以单项目单写者模型作为运行前提，即同一项目目录在同一时刻只允许一个写入执行者
- V1 可以允许串行写入下的 ID 跳号，但不得允许 ID 冲突

### 8.4 设计约束

- 源数据与派生索引必须分离
- 引用关系必须可校验、可清理
- 大规模场景下必须避免让单次常规操作依赖全量扫描

---

## 9. V1 范围与非目标

### 9.1 V1 必做范围

- 项目初始化
- 结构层新增、修改、受控删除
- 功能节点新增与修改
- 失效分析链新增、修改、受控删除
- 基础索引重建
- 一致性校验
- 按组件、AP、严重度、措施状态的查询
- 标准表格导出

### 9.2 V1 非目标

- 多 Agent 并发写入控制
- 跨项目引用和跨项目联动
- 图形化编辑器
- 自动生成高质量 DFMEA 内容的专业推理能力保证
- 复杂权限控制和审批流
- 数据库级事务与锁机制

---

## 10. 风险与关注点

- Markdown 在复杂表格场景下容易出现格式损坏
- 大规模数据下，索引重建和全局扫描成本可能升高
- 如果引用规则过宽，后续校验和删除语义会显著复杂化
- 如果需求文档不先收敛，架构设计容易提前固化错误假设
- 如果派生索引与源数据职责不清，后期会出现多份真相来源
- 如果异常处理要求不明确，后续实现容易出现静默失败或不一致恢复

## 11. 验收标准

当以下验收项全部满足时，可认为 V1 需求被满足：

1. 项目初始化验收
   - 前置条件：空项目目录或新的 DFMEA 项目目标目录
   - 验收动作：执行项目初始化流程
   - 预期结果：生成完整项目基础数据骨架，且项目处于可继续录入状态

2. 结构录入验收
   - 前置条件：项目已初始化
   - 验收动作：录入至少 1 个系统、1 个子系统、1 个组件
   - 预期结果：结构关系可定位、可遍历、可读取，且不存在非法归属

3. 分析录入验收
   - 前置条件：至少存在 1 个组件
   - 验收动作：为该组件录入至少 1 个功能和 1 条完整失效分析链
   - 预期结果：分析对象可查询、可修改、可追溯，且主归属关系有效

4. 查询验收
   - 前置条件：项目中存在结构和分析数据
   - 验收动作：分别执行按组件查询、按 AP 查询、按严重度查询、按措施状态查询、按关键词或 ID 搜索
   - 预期结果：查询结果包含最小必要输出字段，且能定位回源数据

5. 修改与删除验收
   - 前置条件：项目中已存在分析数据和可删除对象
   - 验收动作：对结构节点和分析节点执行合法修改、合法删除及非法删除尝试
   - 预期结果：合法操作成功，非法操作被拒绝并返回清晰原因，删除后的引用关系与一致性状态可被校验

6. 校验验收
   - 前置条件：项目中已存在正常数据，并可人为制造至少一类不一致问题
   - 验收动作：执行一致性校验
   - 预期结果：能够识别并报告归属失效、引用失效、孤儿数据或派生数据失真等问题

7. 索引恢复验收
   - 前置条件：项目中存在派生索引，且索引被置为过期、缺失或失效状态
   - 验收动作：执行索引重建流程
   - 预期结果：横切查询能力恢复，且恢复结果与源数据保持可追溯一致

8. 导出验收
   - 前置条件：项目中存在结构和分析数据
   - 验收动作：执行导出流程
   - 预期结果：导出结果适合审阅，且可以追溯到对应源数据

9. 运行约束验收
   - 前置条件：在本地文件系统和 Windows 环境下运行，且不依赖数据库
   - 验收动作：完成核心使用场景
   - 预期结果：系统在既定约束下可运行，且不要求引入数据库或服务端能力

10. Skill 可执行性验收
   - 前置条件：存在完整需求、架构和 skill 文档
   - 验收动作：由不同 Agent 按文档执行相同基本任务
   - 预期结果：能够得到语义一致、结构一致的基本结果，且不依赖特定 Agent 的隐含能力
